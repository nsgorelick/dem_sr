# DEM Super-Resolution Codebase Design & Implementation Plan

## 1. Purpose

This document defines a code architecture and implementation strategy for the DEM SR project that:

* Maximizes **code reuse** across experiments
* Enables **rapid iteration** on new research plans
* Preserves the current working pipeline
* Avoids over-engineering or premature abstraction

The goal is to support a wide range of experimental directions (plans 01–10) while keeping the system simple, testable, and extensible.

---

## 2. Existing System Overview

### 2.1 Data Extraction

`export_patches_gcs.py`

* Builds patch-aligned stacks from Earth Engine
* Defines:

  * patch geometry
  * band structure (`makeStack`)
  * masks and weights
* Writes:

  * stack GeoTIFFs
  * AE embedding GeoTIFFs

This is already a **strong, centralized data definition layer**.

---

### 2.2 Training

`train_experiment.py`

* Loads patches via `LocalDemPatchDataset`
* Applies augmentation
* Trains FiLM U-Net
* Uses a single loss (`loss_dem`)
* Saves checkpoints with metadata

---

### 2.3 Evaluation

`eval_experiment.py`

* Evaluates multiple prediction sources:

  * model
  * z_lr baseline
  * external rasters
* Computes:

  * weighted elevation metrics
  * slope metrics
  * per-patch metrics
  * stratified metrics via patch table

This is already a **rich evaluation system**.

---

## 3. Design Principles

### 3.1 Primary training / evaluation entrypoints

Training and evaluation run through:

* `train_experiment.py`
* `eval_experiment.py`

`export_patches_gcs.py` remains the stable extraction entrypoint until a v2 export path exists.

New architecture and experiment work should be introduced through:

* new modules under `experiments/`
* shared `train/` and `eval/` engines
* thin CLI wrappers

### 3.2 Keep Scripts Thin

* CLI scripts remain orchestration-only
* Move reusable logic into modules
* New functionality lands in new scripts first

### 3.3 Standardize Interfaces

Define consistent contracts for:

* batch data
* model outputs
* loss functions
* metrics

### 3.4 Prefer Composition Over Inheritance

* Small reusable modules
* Avoid monolithic “plan-specific” code

### 3.5 Optimize for Iteration Speed

* Minimal boilerplate for new experiments
* No heavy framework dependencies
* Preserve working training/eval flows while adding new capability in parallel

## 4. Core Contracts

### 4.1 Data Contract

Each batch should be a dictionary:

```
{
  "x_dem": Tensor,
  "x_ae": Tensor,
  "z_lr": Tensor,
  "z_gt": Tensor,
  "w": Tensor,
  ... optional fields
}
```

Optional fields:

* masks
* candidate rasters
* patch metadata
* auxiliary features

---

### 4.2 Model Contract

Models must return:

```
{
  "z_hat": Tensor,
  ... optional outputs
}
```

Optional outputs:

* residual bands
* uncertainty
* expert weights
* intermediate states

---

### 4.3 Loss Contract

Loss functions accept:

```
loss_fn(outputs, batch)
```

Return:

```
{
  "loss": Tensor,
  "metrics": dict
}
```

---

### 4.4 Metric Contract

Metrics operate on:

* predictions
* ground truth
* weights

Produce:

* scalar metrics
* per-patch metrics
* stratified metrics

## 11. Implementation Plan

### Phase 1 (Immediate, zero disruption)

* Standardize model output conventions in new code only
* Standardize loss output conventions in new code only
* Extract reusable metrics into new shared modules
* Centralize checkpoint helpers in new shared modules
* Do **not** alter `export_patches_gcs.py` without a migration plan for extraction

### Phase 2 (Parallel path)

* Implement composable losses
* Add experiment registry
* Build `train_experiment.py`
* Build `eval_experiment.py`

### Phase 3 (Selective migration)

* Implement new experiments on new entrypoints
* Prefer parity checks on fixed seeds/manifests when changing shared engines

### Phase 4 (Optional extraction modernization)

* Create `export_patches_v2.py` only if needed
* Keep `export_patches_gcs.py` stable unless there is a compelling reason to migrate extraction logic

## 13. Anti-Patterns to Avoid

* Changing shared metric or checkpoint contracts without updating tests
* Per-plan training scripts with duplicated logic
* Monolithic model files
* Hard-coded losses
* Over-general frameworks

---

## 14. Summary

The architecture is built around:

* shared data + model + loss contracts
* modular components
* thin orchestration scripts
* **new entrypoints for all new work**
* **shared engines and contracts covered by unit tests**

This allows rapid experimentation across many research directions while keeping one supported train/eval path.

---

## 15. Current Migration Status

The migration is in **Phase 3+**:

* `train_experiment.py` and `eval_experiment.py` are the supported training/evaluation CLIs
* legacy `train_dem.py` / `eval_dem.py` have been removed
* `train/engine.py` and shared eval helpers are in place
* shared `core/` modules cover:
  * data contract validation
  * checkpoints and resume helpers
  * metrics and reporting payloads
* composable losses are available under `losses/`
* experiment registry supports baseline + placeholder hydrology key

### 15.1 Practical Workflow

* use `train_experiment.py` / `eval_experiment.py`
* use `--preset` and optional `--loss-system composite` for fast setup
* keep run outputs under `runs/` as disposable artifacts

### 15.2 Remaining Work

* replace placeholder hydrology experiment logic with dedicated hydrology model/loss/dataset components
* optional: `export_patches_v2.py` if extraction needs a second path

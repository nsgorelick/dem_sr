# DEM Super-Resolution Codebase Design & Implementation Plan

## 1. Purpose

This document defines a code architecture and implementation strategy for the DEM SR project that:

* Maximizes **code reuse** across experiments
* Enables **rapid iteration** on new research plans
* Preserves the current working pipeline
* Avoids over-engineering or premature abstraction

The goal is to support a wide range of experimental directions (plans 01–10) while keeping the system simple, testable, and extensible.

---

## 2. Existing System Overview

### 2.1 Data Extraction

`export_patches_gcs.py`

* Builds patch-aligned stacks from Earth Engine
* Defines:

  * patch geometry
  * band structure (`makeStack`)
  * masks and weights
* Writes:

  * stack GeoTIFFs
  * AE embedding GeoTIFFs

This is already a **strong, centralized data definition layer**.

---

### 2.2 Training

`train_dem.py`

* Loads patches via `LocalDemPatchDataset`
* Applies augmentation
* Trains FiLM U-Net
* Uses a single loss (`loss_dem`)
* Saves checkpoints with metadata

---

### 2.3 Evaluation

`eval_dem.py`

* Evaluates multiple prediction sources:

  * model
  * z_lr baseline
  * external rasters
* Computes:

  * weighted elevation metrics
  * slope metrics
  * per-patch metrics
  * stratified metrics via patch table

This is already a **rich evaluation system**.

---

## 3. Design Principles

### 3.1 Keep Scripts Thin

* CLI scripts remain entrypoints
* Move reusable logic into modules

### 3.2 Standardize Interfaces

Define consistent contracts for:

* batch data
* model outputs
* loss functions
* metrics

### 3.3 Prefer Composition Over Inheritance

* Small reusable modules
* Avoid monolithic “plan-specific” code

### 3.4 Optimize for Iteration Speed

* Minimal boilerplate for new experiments
* No heavy framework dependencies

---

## 4. Core Contracts

### 4.1 Data Contract

Each batch should be a dictionary:

```
{
  "x_dem": Tensor,
  "x_ae": Tensor,
  "z_lr": Tensor,
  "z_gt": Tensor,
  "w": Tensor,
  ... optional fields
}
```

Optional fields:

* masks
* candidate rasters
* patch metadata
* auxiliary features

---

### 4.2 Model Contract

Models must return:

```
{
  "z_hat": Tensor,
  ... optional outputs
}
```

Optional outputs:

* residual bands
* uncertainty
* expert weights
* intermediate states

---

### 4.3 Loss Contract

Loss functions accept:

```
loss_fn(outputs, batch)
```

Return:

```
{
  "loss": Tensor,
  "metrics": dict
}
```

---

### 4.4 Metric Contract

Metrics operate on:

* predictions
* ground truth
* weights

Produce:

* scalar metrics
* per-patch metrics
* stratified metrics

## 13. Anti-Patterns to Avoid

* Per-plan training scripts
* Monolithic model files
* Hard-coded losses
* Over-general frameworks

---

## 14. Summary

The architecture is built around:

* shared data + model + loss contracts
* modular components
* thin orchestration scripts

This allows rapid experimentation across many research directions while maintaining a stable and reusable codebase.
